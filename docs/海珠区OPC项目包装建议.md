# 海珠区 OPC 方向项目包装建议

## 一、结论

当前项目**可以申报**，但更适合包装为：

- 面向移动/Web 的修仙题材独立游戏产品
- 超级个体模式下的低成本高迭代内容产品
- 游戏场景中的数字内容创新项目

当前项目**不适合直接包装为生成式 AI 项目**，原因是现有工程中没有明确的模型调用、智能体流程、算法备案能力或生成式内容闭环。若强行往 AI 方向靠，会导致材料失真，后续验收风险较高。

因此，现阶段最稳妥的申报路径是：

1. 以“游戏场景项目”申报
2. 以“一人公司 / 超级个体”的低成本研发模式作为加分项
3. 以可运行 Demo、跨端发布能力、持续扩展空间作为项目成熟度证明

## 二、匹配度判断

### 适合的部分

1. **游戏场景明确**
   项目本身就是完整的游戏产品原型，题材、玩法和核心循环都已清晰成立，符合政策里“游戏”为重点场景的方向。

2. **超级个体属性强**
   当前工程结构明显是单人或微型团队推进方式，核心逻辑集中在 `assets/scripts/GrottoExpeditionDemo.ts`，适合包装成“一人公司高效完成内容研发”的案例。

3. **已有可验证成果**
   项目已有 Cocos Creator 工程、Web 发布路径、可运行 Demo、玩法文档和美术工具链，不是停留在概念阶段。

4. **有低成本研发证据**
   仓库内已有素材校验、裁切、资源整理等工具脚本，可证明项目在有限人力下建立了内容生产和资源接入流程，能支撑“降本增效”的叙述。

5. **内容扩展性强**
   当前不仅有主线的百层洞府探秘，还有“独钓万古”等局外玩法扩展，说明项目具备持续更新与商业化延展空间。

### 不适合硬报的部分

1. **不适合报算法备案型项目**
   当前没有可证明的生成式 AI 服务或算法平台能力。

2. **不适合主打算力补贴逻辑**
   工程里没有明显的模型训练、推理成本或 token 消耗场景，和政策里偏 AI 算力支持的部分匹配较弱。

3. **不建议把普通开发工具说成 AI 工具**
   现有的资源裁切和校验脚本属于研发提效工具，但不是智能体或生成式系统，材料里应如实表述为“自动化资源处理流程”。

## 三、建议申报定位

### 推荐定位

**“基于超级个体研发模式的修仙题材轻量内容游戏项目”**

或者：

**“面向移动/Web 场景的修仙 Roguelite 独立游戏产品”**

如果需要更贴近政策语言，可以写成：

**“超级个体模式下的数字互动内容创新与游戏场景落地项目”**

### 不推荐定位

- “生成式 AI 游戏平台”
- “智能体驱动游戏引擎项目”
- “算法备案类互动娱乐系统”

这些表述和当前工程事实不一致。

## 四、当前项目的包装重点

### 1. 把它包装成“产品”，不是“练手 Demo”

外部表述不要强调“只是 Demo”，而要强调：

- 项目已经完成核心玩法闭环验证
- 已具备 Web 端可运行版本
- 已形成局外养成 + 局内探索 + 战斗 + 结算的完整循环
- 已具备后续章节、玩法和商业化扩展能力

### 2. 把单人开发包装成“超级个体效率模型”

要讲清楚的不是“我一个人很辛苦”，而是：

- 单人即可完成玩法设计、程序实现、资源接入和版本迭代
- 项目采用模块化玩法设计和自动化资源处理流程
- 通过脚本化工具降低素材接入和版本调试成本
- 形成适合独立开发者持续迭代的研发范式

### 3. 把资源工具链包装成“降本增效能力”

仓库中的工具脚本可以这样表述：

- 建立了针对美术资源的自动校验机制，减少错误素材进入运行时
- 建立了批量裁切与标准化处理流程，降低 UI 资源返工成本
- 通过规范化资源接入，提升多批次素材替换效率

这部分虽然不是 AI，但属于政策可接受的“降本增效”叙述。

### 4. 把题材和玩法包装成“可传播的内容特色”

项目可突出：

- 修仙题材，有明显东方幻想内容辨识度
- “洞府养成 + 百层探秘 + 搜打撤”玩法组合有一定差异化
- 兼具体验深度与轻量上手特征，适合移动/Web 传播

## 五、可直接使用的一页式项目简介

### 项目名称

《仙人洞府探秘 100 层》

### 申报名称建议

**超级个体模式下的修仙题材互动内容游戏项目**

### 项目简介

《仙人洞府探秘 100 层》是一款由超级个体研发模式推进的修仙题材独立游戏产品，项目围绕“洞府养成 + 百层秘境探索”构建完整玩法闭环，已完成核心系统原型与 Web 端可运行版本验证。玩家在局外经营洞府、提升境界与资源产出，在局内进入秘境逐层探索、战斗、撤离与结算，形成可持续迭代的轻量 Roguelite 体验。

项目采用单人高效率研发方式，结合 Cocos Creator 跨端能力和自动化资源处理工具，完成玩法设计、程序开发、UI 接入与版本迭代，具备低成本研发、快速验证和持续扩展的特点。当前项目已具备后续玩法扩展、内容更新和商业化落地基础，适合作为游戏场景下的数字互动内容创新项目进行培育。

### 核心亮点

1. 修仙题材与百层探秘结合，内容辨识度较强
2. 局外养成与局内探索形成完整体验闭环
3. 单人可持续推进，具备超级个体样板特征
4. 已有 Web 端成果，适合展示、测试和传播
5. 建立了自动化资源处理流程，具备低成本迭代能力

## 六、申报时建议强调的指标

现阶段可以准备以下几类指标：

1. **完成度指标**
   - 已完成核心玩法模块数量
   - 已完成局外、局内、战斗、结算等闭环系统
   - 已上线 Web 演示版本

2. **效率指标**
   - 单人完成从玩法设计到可运行版本的周期
   - 自动化资源处理减少的人工整理工作量
   - 美术素材接入和返修效率提升情况

3. **扩展性指标**
   - 已规划的后续玩法模块数量
   - 可扩展职业、境界、事件、地图层数或系统数
   - 已形成的世界观和玩法文档数量

4. **传播验证指标**
   - Web 演示访问量
   - 玩家测试人数
   - 留存、试玩时长、有效反馈数量

如果暂时没有外部数据，至少应先整理：

- 版本迭代记录
- 可运行视频
- 核心玩法截图
- 开发周期说明

## 七、建议准备的证明材料

1. 一页式项目简介
2. 3 至 5 页项目路演稿
3. 1 至 2 分钟玩法演示视频
4. Web 可试玩链接或本地演示包
5. 核心功能截图
6. 研发流程说明
7. 自动化资源脚本说明
8. 后续版本规划

## 八、建议使用的话术

可以优先使用下面这种表述：

“本项目面向游戏场景，探索超级个体模式下的数字互动内容研发路径。项目以修仙题材轻量 Roguelite 为核心，已完成从玩法设计、程序实现到资源接入和 Web 端验证的完整原型。项目具备低成本研发、高效率迭代和持续扩展能力，适合作为独立内容产品进行培育与落地。”

## 九、需要补强的地方

如果要提高申报成功率，建议补强以下内容：

1. 增加外部试玩和反馈数据
2. 补一版更明确的商业化路径
3. 形成更正式的版本路线图
4. 整理单人研发效率与成本结构说明
5. 若未来确实引入 AI 内容生产，再单独扩展 AI 叙述

## 十、最终判断

当前项目**符合“游戏场景 + 超级个体研发”这一申报方向**，不符合“强 AI / 算法备案 / 大算力消耗”方向。

最合理的包装方式不是夸大 AI，而是强调：

1. 单人完成高完成度修仙内容产品原型
2. 已有可运行 Demo 与跨端展示能力
3. 有自动化资源流程，体现低成本高效率研发
4. 具备持续更新、传播和商业化延展潜力

按这个方向准备材料，逻辑是成立的，且比硬套 AI 叙述更稳。
